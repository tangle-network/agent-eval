## 2. Integration shape

- **Eval entry** — `pnpm eval` → `eval/run.ts:36` → `eval/canonical.ts::runCanonicalEvalCli`. Single CLI flag `--backend cli-bridge|sandbox|tcloud` drives a unified `EvalBackendConfig` (`canonical.ts:286-336`) for agent + simulator + judge + trace-analyst legs; env-var per-leg fallbacks are intentionally hard-cut (`canonical.ts:241-244`, `auto-research.ts:135-152`).
- **Campaign build** — Personas live in a single 32-entry `eval/business-owner/personas.json` (27/32 ship `multiTurnFlow` arrays — real multi-turn coverage). Loaded via `loadPersonas` (`canonical.ts:217-224`). Production-loop holdout scenarios are passed in by the CI cron caller (`production-loop/index.ts:115-116`).
- **Judges** — 12-dimension rubric (`GTM_JUDGE_DIMENSIONS`, `canonical.ts:589-602`) covers `onboardingValidation`, `discoveryQuality`, `integrationReasoning`, `adaptation`, `operatorBriefQuality`, `antiFabrication`, `actionability`, `voiceMatch`, plus 4 operator-CEO dims (`evidenceAnchoring`, `killCriteriaSpecificity`, `competeSpecificity`, `directnessUnderPressure`). Default 3-family ensemble `kimi/glm/deepseek` (`run-prompt-evolution.ts:154-158`). Local `judge-ensemble.ts` enforces no-self-judging via `judgeFamily` regex matching with `JudgeEnsembleError` throws — opinionated, no silent fallback. `judgePersonaRunEnsemble` (`canonical.ts:725-797`) runs all judges in parallel, tolerates partial failures, surfaces `judgeMaxDisagreement` and per-dim mean across survivors.
- **Trace persistence** — Production `runChatThroughRuntime` opens a `TraceEmitter` per chat session (`chat.ts:259-264`), routes `llm_call`/`tool_call`/`tool_result` runtime events into typed substrate spans (`chat.ts:324-358`), and dispatches `onRunComplete` to a product trace sink that composes a `ProductionRunRecord` + OTLP ship. Per-eval-persona `FileSystemTraceStore` under `eval/.runs/<runId>/personas/<id>/trace-store/` (`canonical.ts:1170-1175`); `FileSystemRawProviderSink` at run-root (`canonical.ts:1127-1129`). `assertLlmRoute` preflight (`canonical.ts:1135-1139`) + `assertRunCaptured` post-flight (`canonical.ts:1228-1236`). `agent-eval-traces.ts` exposes a thin wrapper for non-canonical callers.
- **Analyst loop** — `pnpm eval:improve` (`eval/analyst-loop.ts`). 100% substrate-driven: `AnalystRegistry` + `DEFAULT_TRACE_ANALYST_KINDS` + `createTraceAnalystKind` + `FindingsStore` (JSONL, not D1), then delegates orchestration to `@tangle-network/agent-runtime/analyst-loop::runAnalystLoop` with substrate adapters `createSurfaceImprovementAdapter` / `createSurfaceKnowledgeAdapter`. Auto-applies knowledge at ≥0.85 confidence; improvement edits open PRs at ≥0.9 with LLM-drafted unified diffs (`analyst-loop.ts:293-357`). Manifest in `eval/agent.config.ts` declares 5 surfaces (`systemPrompt`, `tools`, `rubric`, `knowledge`, `personas`).
- **Promotion gate** — `eval/run-prompt-evolution.ts:778-918` builds a substrate `pairedEvalueSequence` anytime-valid e-value over (winner − baseline) deltas per persona, round-robin interleaved across scenarios. Stop on `promote_now` (≥1/α), `reject_now`, or budget exhaustion. Decisive verdict drives optional auto-PR open.
- **Calibration** — `pnpm eval:calibrate` (`eval/calibrate-judges.ts`). Per (judge × gold-item × dimension) loop with retry, persists per-judge per-dim raw scores + Pearson via substrate `calibrateJudge`. 12 dim rubric guides mirror `buildJudgeRubric` (`calibrate-judges.ts:53-119`). Null human grades are skipped on that dim only — auditable in `dropped[d]` (line 290-292).
- **Production loop** — `runProductionLoop` is the canonical 5-step entry (ingest → cluster → evolve → gate → PR) wired in `production-loop/index.ts:143-192`. Cluster threshold `minClusterSize: 5`; `ProductionShipConfig` opens PRs on `tangle-network/gtm-agent` via `httpGithubClient`. Worker scheduled handler dispatches the cron string and logs; CI runner actually executes (Worker can't budget-fit the multi-judge × multi-rep matrix).
- **Real product vs synthetic** — Canonical eval drives the REAL `runChatThroughRuntime` in-process (`canonical.ts:81`, `chatTurnThroughRuntime` line 504). Knowledge readiness flags forced true to avoid the production gating (line 525-532). Backend kinds `cli-bridge` (loopback OpenAI-compat), `sandbox` (workspace sandbox + cli-bridge helpers), `tcloud` (router). No synthetic transcript injection.
- **Feedback dataset** — Server records `RecordedFeedbackEvent`s and emits substrate `FeedbackTrajectory` via `createFeedbackTrajectory` (`feedback-dataset.ts:171-232`), then derives optimizer rows via `feedbackTrajectoryToOptimizerRow`. Splits are stable-hashed 70/15/10/5 train/dev/test/holdout (line 65-81) then re-confirmed via `withAssignedFeedbackSplit`. Persisted into the per-workspace vault under `eval-data/{trajectories,optimizer,preferences}-<split>.jsonl.md`. Preference memory rendered via `summarizePreferenceMemory` + `renderPreferenceMemoryMarkdown`.
- **Action policy** — Substrate `evaluateActionPolicy` gates external side-effects (`publish_content`, `send_outreach`) with required kill-criteria + expected-outcome (`action-policy.ts:29-43`). $5 cost-approval threshold, $50 hard cap.

## 4. Drift — open-coded patterns that should be replaced by substrate primitives

- **Composite formula open-coded twice.** `canonical.ts:1209` (`judgeAvg*0.6 + struct*0.3 + slop*0.1`) and `run-prompt-evolution.ts:143-147,460-464`. Should call a substrate composite helper (or at minimum a shared local helper) so the production-loop's MultiShotScorer cannot drift from the canonical scorer.
- **Per-judge × per-dim aggregation re-implemented.** `judgePersonaRunEnsemble` (`canonical.ts:725-797`) computes per-dim mean + max-disagreement manually; substrate's `continuousAgreement` / trial-aggregator could do this off the flat metrics bag. Author flags this themselves (`run-prompt-evolution.ts:516-522`) but never migrates.
- **Auto-research analyst transport built manually.** `runAutoResearch` (`eval/auto-research.ts:122-152`) lazy-imports `@ax-llm/ax` and constructs the AI client; substrate exposes `createTraceAnalystKind` which already handles this — `analyst-loop.ts` uses it correctly, but `auto-research.ts` is the older path and never refactored.
- **OTLP flattening hand-rolled.** `runAutoResearch` (`auto-research.ts:154-179`, called once per run) flattens OTLP array-attribute form into the object shape `OtlpFileTraceStore` expects. Comment admits this is to bridge two substrate surfaces.
- **`scoreDeterministic` regex matchers for kill-criteria / directness / evidence-anchoring** (`canonical.ts:856-916`). These deterministic heuristics duplicate what a tool-call-fidelity matcher in the substrate could express declaratively; they also collide with the LLM judge's view of the same dims (the prompt is told the deterministic output is informational only — `canonical.ts:644`).
- **`splitFor` stable-hash** (`feedback-dataset.ts:65-81`) re-implements feedback split assignment manually before calling `withAssignedFeedbackSplit`. The substrate helper alone would suffice.
- **Banned-phrase list lives in source** (`canonical.ts:189-210`) — should be a curated corpus (knowledge wiki) and consumed by both rubric and deterministic scorer.
- **`hashShort` FNV-1a** (`canonical.ts:1433-1440`) duplicates `hashContent` from the substrate (the smoke test imports `hashContent` but eval/canonical never uses it).

Unique to GTM (worth keeping): the unified `--backend` config (`canonical.ts:286-336`) that hard-cuts per-leg env fallbacks is more disciplined than the parallel verticals. The `JudgeEnsembleError` throw-on-self-judging (`eval/lib/judge-ensemble.ts:51-56,93-125`) is opinionated and well-tested — substrate could absorb this.

## 5. Verdict + 5 highest-leverage upgrades

**Verdict — 7/10.** GTM-agent has the strongest *integration-shape coverage* of the four product verticals: it actually drives the production runtime end-to-end (`runChatThroughRuntime`, not a synthetic transcript), it ships the only TraceEmitter wiring inside the live chat handler (`chat.ts:259-358`), it uses the substrate's production-loop and pairedEvalueSequence promotion gate, and its analyst-loop is purely substrate-adapter-driven. But it lags the 0.31.1 surface materially: no adversarial probes, no tool-call fidelity, no Wilcoxon/bootstrap/Cliff's, no streaming-quality dim, no cost-aware judge selection, no D1-backed FindingsStore despite shipping D1, no prod-trace harvest cron despite shipping wrangler crons. The composite formula and per-judge aggregation are open-coded in two places and quietly drifting.

**Top 5 upgrades (in dependency order, highest leverage first):**

1. **Prod-trace harvest cron + D1-backed FindingsStore.** The production-loop is the substrate's whole point and currently has no pipe-in: `TraceEmitter` writes from the live chat handler, but no cron projects those traces into the `FileSystemTraceStore` the weekly `runProductionLoop` expects. Add a daily Worker cron that flushes traces to D1 (binding `DB` already exists, `worker-configuration.d.ts:3`) and a `D1FindingsStore` shim so `analyst-loop.ts:178-180` gets durable findings across runs. Without this, the analyst loop is local-only and the production-loop is unhittable.
2. **Adversarial probe suite + 0.95 pass-rate gate.** Add 8-12 personas covering prompt-injection (system prompt extraction, banned-phrase smuggling), data-leak (asking the agent to recite persona.badData as fact), tool-misuse (asking for an unsupported integration), refusal-quality (`early-marketing-strategy-refusal` exists but stands alone). Gate `runCanonicalEval` on adversarial-subset pass-rate ≥0.95 before reporting composite; surface as a separate `adversarial.passed` field on `CanonicalEvalSummary` (`canonical.ts:145-172`). Co-locates with existing personas — incremental change.
3. **Tool-call fidelity dim with deterministic matchers.** `chat.ts:336-354` already emits tool spans. Add a 13th rubric dim `toolCallFidelity` keyed off expected matchers declared on each persona (`expectedToolCalls?: { tool: string; argMatcher: ... }[]`). Score via deterministic span comparison (no LLM) — fastest, cheapest signal in the rubric. Personas with `integrationIntents` already declare what tools should be invoked; just close the loop.
4. **Differential A/B: `pairedWilcoxon` + `pairedBootstrap` + Cliff's δ.** `runShipGate` (`run-prompt-evolution.ts:778-918`) reports anytime-valid e-value verdict but nothing else; add Wilcoxon p-value + bootstrap 95% CI + Cliff's δ effect-size to the `ShipGateVerdict` and the markdown PR body. Same paired-deltas input, three more lines, dramatically better reviewer signal — especially when `pairedEvalueSequence` returns `continue`.
5. **Cost-aware Pareto judge selection + real `RunRecord.costUsd`/`wallMs`.** `canonical.ts:1247-1249` hardcodes `wallMs: 0, costUsd: 0, tokenUsage: { input: 0, output: 0 }` and sets `cost_unknown: 1`. The `rawSink` is already capturing per-call usage. Plumb real values into `RunRecord`; then have `calibrate-judges.ts` score (Pearson × marginal cost) and pick the Pareto-frontier 3 judges per run instead of the hard-coded `kimi/glm/deepseek` list (`run-prompt-evolution.ts:154-158`). Closes the substrate's silent-zero anti-pattern that's already flagged in MEMORY.
